# Lead Measure 도메인 설계

## 1. 개요

**선행지표(Lead Measure)**는 WIG 달성을 위해 사용자가 **직접 통제할 수 있는 행동 지표**다.  
4DX 원칙에 따라, 좋은 선행지표는 "예측적(predictive)"이고 "통제 가능(influenceable)"해야 한다.

---

## 2. 핵심 규칙 (Business Rules)

| # | 규칙 |
|---|------|
| BR-1 | 선행지표는 ACTIVE 상태의 점수판에만 추가/수정/삭제 가능 |
| BR-2 | 지표 상태: `ACTIVE` (집계 포함) / `ARCHIVED` (집계 제외) |
| BR-3 | 물리 삭제 시 해당 지표의 모든 Daily Log도 함께 삭제 |
| BR-4 | 보관(Archive) 시 `archived_at` 기록, 이후 Daily Log 추가 불가 |
| BR-5 | 재활성화(Reactivate) 시 `status: ACTIVE`, `archived_at: null` 로 초기화 |
| BR-6 | 점수판당 선행지표 최대 개수 제한 없음 (UX 가이드로 5개 이내 권장) |

---

## 3. 4DX 가이드 UX

지표 입력 폼에서 사용자가 좋은 선행지표를 작성할 수 있도록 단계별 가이드 제공:

| 단계 | 질문 | 예시 |
|------|------|------|
| 행동 정의 | "무엇을 할 것인가?" | "매일 물 2L 마시기" |
| 예측성 검토 | "이 행동이 WIG 달성에 기여하는가?" | ✅ / ⚠️ 아이콘으로 피드백 |
| 통제 가능성 검토 | "내가 직접 통제할 수 있는가?" | ✅ / ⚠️ 아이콘으로 피드백 |
| 수치화 | "목표 빈도는?" | 매일(DAILY) / 주 N회 |

---

## 4. 측정 방식

| 타입 | 설명 | 예시 |
|------|------|------|
| `DAILY` | 매일 O/X 기록 | "매일 물 2L 마시기" |
| `WEEKLY` | 주간 목표 빈도 설정 (예: 주 3회) | "주 3회 운동" |

> **MVP 범위**: DAILY 타입 중심으로 구현. WEEKLY는 `target_value` 필드로 집계.

---

## 5. 데이터 모델

```typescript
interface LeadMeasure {
  id: string;
  scoreboard_id: string;
  name: string;                         // 지표명 (예: "매일 물 2L 마시기")
  target_value: number;                 // 목표 빈도 (1=매일, 3=주3회 등)
  period: 'DAILY' | 'WEEKLY';          // 측정 주기
  status: 'ACTIVE' | 'ARCHIVED';
  created_at: string;
  archived_at: string | null;
}
```

## 6. API 명세

### GET `/api/scoreboards/:scoreboardId/lead-measures`

**Query Params**
- `status`: `active` | `all` (기본값: `active`)

**Response (200 OK)**
```json
[
  {
    "id": "lm_uuid",
    "name": "매일 물 2L 마시기",
    "targetValue": 7,
    "period": "DAILY",
    "status": "ACTIVE",
    "archivedAt": null,
    "weeklyAchievement": { "achieved": 5, "total": 7 }  // 이번 주 달성 현황
  }
]
```

---

### POST `/api/scoreboards/:scoreboardId/lead-measures`

**Request**
```json
{
  "name": "매일 물 2L 마시기",
  "targetValue": 7,
  "period": "DAILY"
}
```

**Response (201 Created)** — 생성된 LeadMeasure 객체

**Error Cases**
| 상태 코드 | 사유 |
|-----------|------|
| 403 | ARCHIVED 점수판에 추가 시도 |
| 400 | 필수 필드 누락 |

---

### PUT `/api/lead-measures/:id`

**Request** — 수정할 필드만 포함
```json
{ "name": "매일 물 1.5L 마시기" }
```

**Error Cases**
| 상태 코드 | 사유 |
|-----------|------|
| 403 | ARCHIVED 점수판의 지표 수정 시도 |

---

### POST `/api/lead-measures/:id/archive`

> 지표 보관 (집계 제외, Daily Log 추가 불가)

**Response (200 OK)**
```json
{ "id": "lm_uuid", "status": "ARCHIVED", "archivedAt": "2026-03-12T00:00:00Z" }
```

---

### POST `/api/lead-measures/:id/reactivate`

> 보관된 지표 재활성화

**Response (200 OK)**
```json
{ "id": "lm_uuid", "status": "ACTIVE", "archivedAt": null }
```

---

### DELETE `/api/lead-measures/:id`

> 물리적 삭제 — 연결된 모든 Daily Log 함께 삭제

**Response (200 OK)**
```json
{
  "warning": "삭제된 기록은 복구할 수 없습니다. 17개의 기록이 함께 삭제됩니다.",
  "deleted": true
}
```

> 클라이언트에서 반드시 확인 다이얼로그 노출 후 호출.

---

## 7. UI 가이드라인

- **추가 버튼**: 점수판 하단에 `+ 선행지표 추가` 버튼 항상 노출
- **정렬**: 생성 순서대로 정렬 (ARCHIVED 지표는 별도 섹션 접기)
- **주간 달성 게이지**: 각 지표 오른쪽에 `5/7` 형식의 달성 현황 표시
- **삭제 전 경고**: "N개의 기록이 영구 삭제됩니다" 카운트 표시
