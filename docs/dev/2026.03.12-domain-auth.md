# Auth 도메인 설계

## 1. 개요

WIG는 **관리자(ADMIN)가 직접 계정을 생성**하고 배포하는 폐쇄형 인증 방식을 채택한다.  
일반 회원가입(Self-signup)은 제공하지 않으며, ID/Password 기반으로 로그인한다.

---

## 2. 핵심 규칙 (Business Rules)

| # | 규칙 |
|---|------|
| BR-1 | 계정 생성은 ADMIN 권한의 사용자만 가능 |
| BR-2 | 초기 비밀번호는 `user`로 일괄 설정 |
| BR-3 | 최초 로그인 시, 비밀번호 변경 권장 토스트 1회 노출 |
| BR-4 | 비밀번호 변경은 강제가 아닌 권장 (강제 이탈 없음) |
| BR-5 | 세션은 JWT 기반. 만료 시 재로그인 유도 |
| BR-6 | 로그아웃 시 클라이언트 토큰 삭제 |
| BR-7 | ID/PW 유효성 검사는 **클라이언트 + 서버 양쪽**에서 수행 (Defense in Depth) |

---

## 2-1. ID / PW 유효성 규칙 (Validation Rules)

### ID (`custom_id`)

| 항목 | 규칙 |
|------|------|
| 허용 문자 | 영문 소문자/대문자, 숫자 (`a-z`, `A-Z`, `0-9`) |
| 금지 문자 | 특수문자 전체, 공백 |
| 길이 | 3자 이상 ~ 20자 이하 |
| 정규식 | `^[a-zA-Z0-9]{3,20}$` |

```typescript
const ID_REGEX = /^[a-zA-Z0-9]{3,20}$/;
export const validateCustomId = (id: string): boolean => ID_REGEX.test(id);
```

---

### PW (`password`)

| 항목 | 규칙 |
|------|------|
| 최소 길이 | 8자 이상 |
| 허용 문자 | 영문, 숫자, 안전 특수문자 |
| **허용 특수문자** | `! @ # $ % ^ & * ( ) - _ = + [ ] { } \| : < > ? , . / ~` |
| **금지 특수문자** | `'` `"` `` ` `` `;` `\` — SQL Injection / 파싱 취약점 유발 가능 |
| 정규식 | `^[a-zA-Z0-9!@#$%^&*()\-_=+\[\]{}|:<>?,./~]{8,}$` |

```typescript
// 허용: 영문, 숫자, 안전 특수문자
// 금지: ' " ` ; \ (SQL injection / 파싱 취약점 유발 문자)
const PW_REGEX = /^[a-zA-Z0-9!@#$%^&()\-_=+\[\]{}|:<>?,./~]{8,}$/;
export const validatePassword = (pw: string): boolean => PW_REGEX.test(pw);
```

> **중요**: SQL Injection의 근본적 방어는 **Prepared Statement(파라미터 바인딩)**로 처리한다. 위 문자 제한은 추가 방어 레이어(Defense in Depth)이며, 문자 제한만으로 SQL Injection을 막는다고 가정하지 않는다.

> **초기 비밀번호 예외**: 기본값 `user`는 위 규칙을 충족하지 않는다. 비밀번호 **변경 시에만** 유효성 검사를 적용하고, 로그인 시에는 해시 비교만 수행한다.

## 3. 사용자 흐름 (User Flow)

### 3.1. 일반 로그인
```
[로그인 화면]
  └─ ID / PW 입력
  └─ 로그인 버튼 클릭
      ├─ 성공 → JWT 발급 → 대시보드 이동
      │         └─ (최초 로그인 여부 확인)
      │             └─ 최초 O → 비밀번호 변경 권장 토스트 노출
      └─ 실패 → "아이디 또는 비밀번호가 올바르지 않습니다" 에러 메시지
```

### 3.2. 관리자 계정 생성 (Admin Only)
```
[설정 > 멤버 관리] (ADMIN 한정)
  └─ "새 멤버 추가" 버튼
  └─ ID / 닉네임 입력
  └─ 저장 → 초기 비밀번호 `user`로 계정 생성
  └─ 생성된 ID를 복사하여 팀원에게 전달 (초대 코드 별도 아님)
```

## 4. 데이터 모델

```typescript
// users 테이블 연관 필드
interface User {
  id: string;              // UUID (PK)
  custom_id: string;       // 로그인용 ID (예: 'john123') — UNIQUE
  password_hash: string;   // bcrypt hash
  nickname: string;
  is_first_login: boolean; // 최초 로그인 여부 추적 → 토스트 제어
  created_at: string;
}
```

> **참고**: `is_first_login` 필드는 DB 스키마에 추가 필요 (현재 schema에 미포함).

## 5. API 명세

### POST `/api/auth/login`

**Request**
```json
{ "customId": "john123", "password": "user" }
```

**Response (200 OK)**
```json
{
  "token": "<JWT>",
  "user": {
    "id": "uuid",
    "nickname": "존",
    "isFirstLogin": true
  }
}
```

**Error Cases**
| 상태 코드 | 사유 |
|-----------|------|
| 401 | 아이디 또는 비밀번호 불일치 |
| 400 | 입력값 누락 |

---

### POST `/api/auth/logout`

- 클라이언트 토큰 삭제 (서버는 stateless이므로 서버 측 무효화는 없음)
- **Response**: `204 No Content`

---

### PUT `/api/auth/password`

> 인증 필요 (Bearer Token)

**Request**
```json
{ "currentPassword": "user", "newPassword": "newSecurePass1!" }
```

**비밀번호 유효성 규칙** — 섹션 2-1 참조
- 최소 8자 이상
- 허용 특수문자: `! @ # $ % ^ & * ( ) - _ = + [ ] { } | : < > ? , . / ~`
- **금지 특수문자**: `'` `"` `` ` `` `;` `\` (SQL Injection 취약점 유발 가능)

**Response (200 OK)**
```json
{ "message": "비밀번호가 변경되었습니다." }
```

**Error Cases**
| 상태 코드 | 사유 |
|-----------|------|
| 400 | 현재 비밀번호 불일치 |
| 422 | 비밀번호 유효성 규칙 미충족 |

---

### POST `/api/admin/users` (ADMIN Only)

> ADMIN 권한 인증 필요

**Request**
```json
{ "customId": "newmember", "nickname": "새멤버" }
```

**Response (201 Created)**
```json
{
  "id": "uuid",
  "customId": "newmember",
  "nickname": "새멤버",
  "initialPassword": "user"
}
```

## 6. 엣지 케이스

| 케이스 | 처리 방식 |
|--------|----------|
| 이미 사용 중인 ID로 계정 생성 시도 | `409 Conflict` 반환 |
| 만료된 JWT로 API 호출 | `401 Unauthorized` → 클라이언트 로그인 화면으로 리다이렉트 |
| 비밀번호 변경 후 기존 토큰 계속 사용 | 기존 토큰 유효 유지 (보안 강화는 Post-MVP) |

---

## 7. 보안 고려사항

- 비밀번호는 **bcrypt** (cost factor 12+) 해시 후 저장
- JWT 만료: `24h` (MVP), 리프레시 토큰은 Post-MVP
- 로그인 실패 시 구체적인 사유 미노출 (ID/PW 중 어느 것이 틀렸는지 구분하지 않음)
- HTTPS 전용 환경 (Cloudflare)
