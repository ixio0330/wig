## 2. 현재 구현 사실 (코드 기준)

### 2.1. 제공 API

- `GET /api/workspaces/me`
- `POST /api/workspaces` (생성)
- `POST /api/workspaces/join` (ID 기반 참가)
- `PUT /api/workspaces/:id` (이름 변경, ADMIN)
- `GET /api/workspaces/:id/members` (멤버 목록, ADMIN)
- `DELETE /api/workspaces/:id/members/:memberId` (멤버 퇴출, ADMIN)

### 2.2. 현재 비즈니스 규칙

- 생성/참가 시 `findUserWorkspace(userId)`로 "이미 소속됨" 체크
- 이미 소속이면 `ALREADY_IN_WORKSPACE`(409)
- 멤버 퇴출 시:
  - 자기 자신 퇴출 금지
  - 마지막 ADMIN 퇴출 금지 (`CANNOT_REMOVE_LAST_ADMIN`)
- 권한 검증은 `requireWorkspaceAdminInWorkspace`로 처리

### 2.3. 데이터 모델 특징

- `workspace_members`는 다대다 구조 자체는 이미 지원 가능
- 하지만 현재 서비스 로직은 1인 1워크스페이스를 전제로 동작
- DB 레벨에서 `workspace_members.user_id` unique 제약은 없음

## 3. 문서/스펙과의 차이

- 도메인 설계 문서에 있던 "워크스페이스 삭제"는 현재 API 미구현
- OpenAPI의 `/workspaces/join` 응답 스키마는 `Workspace`인데, 실제 응답은 `{ message }`
- `/workspaces/join` 실패 코드도 스펙(400)과 구현(주로 409/500 가능)이 다르다.
- 존재하지 않는 workspaceId 참가 시 서비스에서 선확인하지 않아 DB FK 에러가 `INTERNAL_ERROR`로 떨어질 여지가 있다.

## 4. 변경 목표 (범용 서비스 + 무료 가치 우선)

### 4.0. 확정 결정 (2026-03-24)

- 가입은 Auth 공개 signup에서 처리하고, Workspace는 초대코드 기반 합류를 기본 경로로 사용한다.
- 계정 복구는 Auth의 복구코드가 기본이며, Workspace ADMIN의 수동 비밀번호 초기화를 보조 수단으로 둔다.

### 4.1. 목표

- 신규 사용자가 워크스페이스 생성/참가를 셀프서브로 완료할 수 있어야 한다.
- 초대 흐름(코드/링크)으로 팀 온보딩 마찰을 낮춰야 한다.
- 다중 워크스페이스 지원 여부를 정책으로 확정하고, 코드/스키마/응답을 일치시켜야 한다.

### 4.2. 비목표

- 결제 플랜 권한 과금 연동
- 대규모 조직용 고급 권한 모델

### 4.3. 프론트 온보딩 분기 기준 (확정)

- 로그인/회원가입 직후 `GET /api/workspaces/me` 호출
- `200`: 기존 대시보드 진입
- `404`: 워크스페이스 없음 온보딩 화면 진입
  - `워크스페이스 만들기` CTA
  - `초대코드로 참가하기` CTA

## 5. 변경안 (백엔드)

### 5.1. 참가 흐름 안정화 (필수)

- `joinWorkspace`에서 대상 워크스페이스 존재 여부를 먼저 검증
- 미존재 시 `NOT_FOUND` 명시 반환 (현재 잠재 500 제거)
- `/workspaces/join` 응답 계약을 OpenAPI와 실제 구현 중 하나로 통일

### 5.2. 초대 흐름 도입 (권장)

- 초대 엔티티(예: `workspace_invites`) 추가 검토
  - `code`, `workspaceId`, `maxUses`, `usedCount`, `status(ACTIVE|INACTIVE)`, `createdBy`
- API 초안:
  - `POST /api/workspaces/:id/invites` (ADMIN)
  - `PATCH /api/workspaces/:id/invites/:inviteId/status` (ADMIN, 활성/비활성 토글)
  - `POST /api/workspaces/join-by-invite`
- 기존 `workspaceId 직접 입력` 방식은 내부/관리자용으로 축소 가능
- 생성 규칙:
  - 초대코드는 `사용 횟수 제한(maxUses)` 기반으로만 발급한다.
  - 만료 시간은 현재 범위에서 사용하지 않는다.
  - 만료 시간(expiration) 기반 정책은 현재 우선순위에서 제외한다.

### 5.5. 관리자 수동 복구 연계 (보조)

- 같은 워크스페이스 ADMIN이 멤버 비밀번호 초기화를 수행할 수 있는 관리자 API를 제공한다.
- 복구코드 사용이 어려운 사용자에 대한 운영 보조 경로로만 사용한다.

### 5.3. 다중 워크스페이스 정책 확정

- 옵션 A: 1인 1워크스페이스 유지
  - DB unique 제약 추가로 무결성 강화
- 옵션 B: 다중 워크스페이스 허용(범용 서비스 권장)
  - `findUserWorkspace` 단일 반환 의존 코드 제거
  - "현재 활성 워크스페이스" 개념 도입 필요
- 현재 결정:
  - 당분간 옵션 A를 유지한다.
  - 다중 워크스페이스는 실제 운영 신호가 쌓일 때까지 후순위로 둔다.

### 5.4. 다중 워크스페이스 허용 시 백엔드 변경 포인트

- 조회 API 확장:
  - `GET /api/workspaces/me` -> 목록형 또는 `GET /api/workspaces` 신설
- 컨텍스트 선택:
  - 헤더/쿼리/세션 기반 active workspace 지정 정책
- authz 헬퍼:
  - 현재 사용자-워크스페이스 멤버십 기준 검증 유지
