# Workspace 백엔드 구현 결과

## 1. 구현 개요

워크스페이스 도메인의 핵심 비즈니스 로직과 API를 **Backend TDD 워크플로우**에 따라 구현 완료하였습니다. 초기 설계에 포함되었던 초대 코드(Invite Code)는 사용자 요청에 따라 스펙에서 제외하고, 워크스페이스 ID 기반으로 동작하도록 단순화되었습니다.

## 2. 주요 작업 내역

### 2.1. 데이터베이스 스키마 및 Drizzle 설정
- `workspaces`: 워크스페이스 기본 정보 테이블 (name, created_at).
- `workspace_members`: 워크스페이스 소속 멤버 매핑 테이블 (workspace_id, user_id, role, privacy_level).
- `src/db/schema.ts`에 관계 설정(Relations) 포함하여 반영 완료.

### 2.2. 도메인 레이어 구현 (TDD)
- **Storage (`WorkspaceStorage`)**: Drizzle ORM을 사용한 DB 접근 레이어.
  - `findUserWorkspace`, `createWorkspace`, `addMember`, `findMembers` 구현.
- **Service (`WorkspaceService`)**: 비즈니스 로직 및 에러 핸들링 레이어.
  - 워크스페이스 생성 시 생성자를 `ADMIN`으로 자동 지정.
  - 중복 워크스페이스 가입 방지 로직 포함 (`ConflictError`).

### 2.3. API 라우트 구현
- `GET /api/workspaces/me`: 내 워크스페이스 정보 조회.
- `POST /api/workspaces`: 새 워크스페이스 생성 (ADMIN 권한 부여).
- `POST /api/workspaces/join`: 워크스페이스 참가.
- `GET /api/workspaces/:id/members`: 워크스페이스 멤버 목록 조회.

## 3. 유효성 검사 (Zod)
- `src/domain/workspace/validation.ts`에 요청 스키마 정의.
  - `workspaceCreateSchema`: 이름 필수값 검증.
  - `workspaceJoinSchema`: 숫자 형태의 ID 검증.

## 4. 검증 결과

### 4.1. 단위 및 통합 테스트
- `WorkspaceStorage`: 4개 테스트 케이스 통과.
- `WorkspaceService`: 4개 테스트 케이스 통과.
- `tsc`: 타입 오류 없음 확인.

### 4.2. 일관성 유지
- 공통 에러 객체(`lib/errors.ts`) 및 응답 규격(`apiSuccess`, `apiError`) 준수.
