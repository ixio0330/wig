# 가중목 서비스 기획 개요

## 1. 배경 및 맥락 (Background & Context)

- **배경**: 현재 지인들과 가중목(WIG)을 관리하며 주 1회 회의를 하고 있음. 노션(Notion)을 통해 기록하고 있으나, 시스템의 능동성이 부족함.
- **문제**: 노션은 직접 찾아가서 채워야 하는 구조라, 일상에 쫓기다 보면 **모임 직전에 다급하게 점수판을 채워넣는** 나를 발견하게 됨. 목표가 단지 기록을 위한 기록으로 전락하고, 스스로 동기부여가 되지 않아 관성적으로 참여하고 있음.

## 2. 우선순위 및 당위성 (Priority & Rationale)

- **왜 지금 해야 하는가?**: 단순히 '기록'하는 수준을 넘어, 4DX를 제대로 활용하여 삶의 실질적인 변화를 만들어내고 싶음. 스스로 동기부여가 되지 않는 현재의 '흐지부지한' 상태를 혁파해야 함.
- **다른 것이 아닌 이것을 왜 해야 하는가?**: 범용적인 도구(노션)는 4DX의 핵심 철학인 '집중'과 '실시간 가시성'을 보장해주지 못함. 내가 매일 기분 좋게 들여다보고, 나를 자극하는 전용 시스템이 필요함.

## 3. 기대 효과 (Impact: Before & After)

- **이전**: 모임 직전 숙제하듯 점수판을 채움. 동기부여가 안 되어 목표가 표류함. 지인들 틈에 섞여 관성적으로 흘러가는 느낌.
- **이후**: '항상 옆에 있는 점수판'을 통해 매일 나의 진척도를 즐겁게 확인. 억지로 채우는 기록이 아닌, 나 스스로의 승리를 확인하는 과정으로 변화하여 내적인 몰입도가 극대화됨.

## 4. 상세 설계 (Detailed Design)

### 4.1. 데이터 모델 (Data Model)

- **User (사용자)**: 닉네임과 수동 ID 기반의 인증 주체.
- **Workspace (워크스페이스)**: 팀원들이 모이는 전용 공간 (MVP는 1인 1워크스페이스).
- **Scoreboard (점수판)**: `목표 (WIG) > 후행지표 (Lag) > 선행지표 (Lead)`의 집합체.
- **Daily Log (일일 기록)**: 선행지표에 대한 요일별 O/X 성취 기록.

### 4.2. UI/UX 흐름 (UI/UX Flow)

- **Dashboard**: "이번 주 나의 승패"를 직관적으로 노출. WIG 게이지와 이번 주 선행지표 현황을 한 줄로 요약.
- **Meeting Mode**: 15분 회의를 위한 공약 체크, 결과 확인, 차주 공약 입력이 한 화면에서 이루어지는 흐름. 팀 단위로 사용 시 돌아가며 보고할 수 있는 뷰 제공.

### 4.3. 기간 탐색형 My Dashboard 확장안 (2026-03-16)

- **문제**: 현재 `dashboard/my`의 핵심 점수판은 항상 현재 주만 보여주기 때문에, 사용자가 특정 주의 회고를 하거나 월간 추세를 한 번에 확인하기 어렵다.
- **왜 지금 필요한가**: 실제 구현에서는 이미 주간/월간 달성률 요약이 함께 노출되고 있어, 본문 테이블과 상단 요약의 시간 기준이 분리되어 보인다. 기간 선택 기능이 없으면 요약 수치를 해석하기도 어렵다.
- **범위**: 우선 `dashboard/my`에만 적용한다. 팀 뷰는 기존처럼 현재 주 중심을 유지하고, 기간 탐색은 Post-MVP 후보로 분리한다.
- **MVP 방향**:
  - 사용자가 기준 날짜를 선택할 수 있어야 한다.
  - 선택된 날짜를 기준으로 `주간` 또는 `월간` 모드를 전환할 수 있어야 한다.
  - 주간 모드에서 날짜 선택은 "그 날짜가 속한 주"를 고르는 용도여야 하며, 결과는 항상 ISO week(월~일) 7일 단위로 보여준다.
  - 월간 모드에서는 해당 월 전체를 기준으로 월간 달성률과 선행지표별 집계표를 보여준다.
  - 월간 모드에서는 일별 토글 테이블 대신 월 집계 중심 UI를 우선 적용하고, 월 내 일별 상세 drill-down은 후속 범위로 둔다.
- **사용자 경험 원칙**:
  - 기본 진입은 여전히 "오늘이 속한 주"다.
  - 주간 모드에서는 일자를 하나 선택하더라도 단일 날짜 상세 뷰로 내려가지 않는다.
  - 사용자가 과거/미래 주를 탐색해도 "오늘" 강조는 현재 실제 날짜에만 유지한다.
  - 기준 날짜와 보기 단위(`week` / `month`)는 URL query에 반영해 새로고침과 공유 시 상태를 복원할 수 있어야 한다.
  - 월간 모드에서 월 단위 지표와 주간 전용 지표를 혼합해 오해를 만들지 않도록, 지표 period에 맞는 집계 기준을 명시한다.

## 7. 성공 및 실패 판단 기준 (Success & Failure Criteria)

- **성공**: 지인들과의 15분 가중목 회의 시, 스프레드시트 대신 이 서비스만 사용하여 보고와 계획이 완결될 때.
- **실패**: 점수판을 보고도 현재 내가 잘하고 있는지(Winning) 즉각적으로 파악되지 않을 때.

## 8. 구현 메모 (2026-03-15)

- `dashboard/my`는 현재 프로토타입 UI를 유지한 채 실제 세션 쿠키 기반 API를 사용한다.
- 사용자 닉네임 표시는 세션 기반 API 조회 결과를 기준으로 표시한다.
- 변경 파일 기준 ESLint는 통과했다.
- 저장소 전체 `yarn tsc --noEmit`는 이번 작업과 별개인 기존 타입 오류로 아직 통과하지 않는다.

## 9. 기간 선택형 대시보드 성공 기준 (2026-03-16)

- 사용자가 `dashboard/my`에서 임의 날짜를 골라 해당 주 또는 해당 월의 지표를 즉시 확인할 수 있다.
- 주간 모드에서는 어떤 날짜를 고르더라도 결과가 항상 1일치가 아닌 주간 7일 테이블로 일관되게 표시된다.
- 상단 요약 카드와 본문 테이블/리스트가 항상 같은 기간 기준으로 동기화된다.
- URL 공유 또는 새로고침 이후에도 선택한 기간과 보기 단위가 유지된다.
- 월간 모드에서 "이번 달 달성률"이 아니라 "선택한 월 달성률"로 의미가 명확하게 전달된다.
