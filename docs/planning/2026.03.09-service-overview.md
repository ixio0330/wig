# 가중목 서비스 기획 개요

## 1. 배경 및 맥락 (Background & Context)

- **배경**: 현재 지인들과 가중목(WIG)을 관리하며 주 1회 회의를 하고 있음. 노션(Notion)을 통해 기록하고 있으나, 시스템의 능동성이 부족함.
- **문제**: 노션은 직접 찾아가서 채워야 하는 구조라, 일상에 쫓기다 보면 **모임 직전에 다급하게 점수판을 채워넣는** 나를 발견하게 됨. 목표가 단지 기록을 위한 기록으로 전락하고, 스스로 동기부여가 되지 않아 관성적으로 참여하고 있음.

## 2. 우선순위 및 당위성 (Priority & Rationale)

- **왜 지금 해야 하는가?**: 단순히 '기록'하는 수준을 넘어, 4DX를 제대로 활용하여 삶의 실질적인 변화를 만들어내고 싶음. 스스로 동기부여가 되지 않는 현재의 '흐지부지한' 상태를 혁파해야 함.
- **다른 것이 아닌 이것을 왜 해야 하는가?**: 범용적인 도구(노션)는 4DX의 핵심 철학인 '집중'과 '실시간 가시성'을 보장해주지 못함. 내가 매일 기분 좋게 들여다보고, 나를 자극하는 전용 시스템이 필요함.

## 3. 기대 효과 (Impact: Before & After)

- **이전**: 모임 직전 숙제하듯 점수판을 채움. 동기부여가 안 되어 목표가 표류함. 지인들 틈에 섞여 관성적으로 흘러가는 느낌.
- **이후**: '항상 옆에 있는 점수판'을 통해 매일 나의 진척도를 즐겁게 확인. 억지로 채우는 기록이 아닌, 나 스스로의 승리를 확인하는 과정으로 변화하여 내적인 몰입도가 극대화됨.

## 4. 상세 설계 (Detailed Design)

### 4.1. 데이터 모델 (Data Model)

- **User (사용자)**: 닉네임과 수동 ID 기반의 인증 주체.
- **Workspace (워크스페이스)**: 팀원들이 모이는 전용 공간 (MVP는 1인 1워크스페이스).
- **Scoreboard (점수판)**: `목표 (WIG) > 후행지표 (Lag) > 선행지표 (Lead)`의 집합체.
- **Daily Log (일일 기록)**: 선행지표에 대한 요일별 O/X 성취 기록.

### 4.2. UI/UX 흐름 (UI/UX Flow)

- **Dashboard**: "이번 주 나의 승패"를 직관적으로 노출. WIG 게이지와 이번 주 선행지표 현황을 한 줄로 요약.
- **Meeting Mode**: 15분 회의를 위한 공약 체크, 결과 확인, 차주 공약 입력이 한 화면에서 이루어지는 흐름. 팀 단위로 사용 시 돌아가며 보고할 수 있는 뷰 제공.

### 4.3. 기간 탐색형 My Dashboard 확장안 (2026-03-16)

- **문제**: 현재 `dashboard/my`의 핵심 점수판은 항상 현재 주만 보여주기 때문에, 사용자가 특정 주의 회고를 하거나 월간 추세를 한 번에 확인하기 어렵다.
- **왜 지금 필요한가**: 실제 구현에서는 이미 주간/월간 달성률 요약이 함께 노출되고 있어, 본문 테이블과 상단 요약의 시간 기준이 분리되어 보인다. 기간 선택 기능이 없으면 요약 수치를 해석하기도 어렵다.
- **범위**: 우선 `dashboard/my`에만 적용한다. 팀 뷰는 기존처럼 현재 주 중심을 유지하고, 기간 탐색은 Post-MVP 후보로 분리한다.
- **MVP 방향**:
  - 사용자가 기준 날짜를 선택할 수 있어야 한다.
  - 선택된 날짜를 기준으로 `주간` 또는 `월간` 모드를 전환할 수 있어야 한다.
  - 주간 모드에서 날짜 선택은 "그 날짜가 속한 주"를 고르는 용도여야 하며, 결과는 항상 ISO week(월~일) 7일 단위로 보여준다.
  - 월간 모드에서는 해당 월 전체를 기준으로 월간 달성률과 선행지표별 집계표를 보여준다.
  - 월간 모드에서는 일별 토글 테이블 대신 월 집계 중심 UI를 우선 적용하고, 월 내 일별 상세 drill-down은 후속 범위로 둔다.
- **사용자 경험 원칙**:
  - 기본 진입은 여전히 "오늘이 속한 주"다.
  - 주간 모드에서는 일자를 하나 선택하더라도 단일 날짜 상세 뷰로 내려가지 않는다.
  - 사용자가 과거/미래 주를 탐색해도 "오늘" 강조는 현재 실제 날짜에만 유지한다.
  - 기준 날짜와 보기 단위(`week` / `month`)는 URL query에 반영해 새로고침과 공유 시 상태를 복원할 수 있어야 한다.
  - 월간 모드에서 월 단위 지표와 주간 전용 지표를 혼합해 오해를 만들지 않도록, 지표 period에 맞는 집계 기준을 명시한다.

### 4.4. `dashboard/my` 기간·지표 선택형 Export 확장안 (2026-03-18)

- **문제**: 사용자가 대시보드에서 보는 기간별 지표를 회고/회의 자료로 저장하려면 수작업 복사가 필요하다.
- **왜 지금 필요한가**: 현재 `dashboard/my`는 기간 탐색(주간/월간 + 기준 날짜)이 가능해졌고, 선택한 기간 기준으로 데이터를 외부에서 재사용할 수 있어야 사용자 가치가 완성된다.
- **운영 제약**: Cloudflare Workers Free 플랜 환경에서는 서버에서 `xlsx`를 직접 생성하는 방식이 CPU time 제약에 걸릴 위험이 높다.
- **MVP 방향(옵션 1 채택)**:
  - 서버는 파일을 만들지 않고, 선택 조건에 맞는 원본/집계 데이터를 반환한다.
  - 프론트엔드는 서버 응답을 기반으로 `csv` 또는 `xlsx` 파일을 생성해 다운로드한다.
  - export 대상은 `dashboard/my`에서 사용자가 직접 선택한 선행지표와 기간(`from`, `to`)으로 한정한다.
  - 팀 일괄 export는 개인정보/권한 정책 확정 전까지 제외한다.
- **입력 UX 원칙**:
  - 기간 입력: `from` / `to` 날짜 필수, `from <= to` 검증.
  - 기간 길이: `from`~`to`는 최대 92일(약 3개월)까지만 허용한다.
  - 지표 입력: `전체 선행지표` 또는 `선택 지표`(복수 선택) 지원.
  - 기간 기본값: 현재 보기 단위 기준으로 자동 채움(주간이면 해당 주, 월간이면 해당 월).
- **데이터 반환 원칙(서버)**:
  - 화면 장식용 문구가 아니라 파일 재가공 가능한 구조화 데이터 제공.
  - 결과에는 최소 `periodMeta`, `summary`, `leadMeasureBreakdown`, `dailyRows` 포함.
  - 과도한 대용량 응답을 피하기 위해 최대 기간 92일 제한과 페이지네이션/분할 다운로드 전략을 둔다.
- **파일 생성 원칙(프론트)**:
  - 기본 포맷은 `csv` 우선(호환성/성능), `xlsx`는 후속 옵션으로 둔다.
  - 파일명은 검색 조건이 드러나도록 `wig-my-export-YYYYMMDD-YYYYMMDD.csv` 패턴 사용.
  - 생성 실패 시 재시도/토스트 에러를 제공하고, API 재호출 없이 동일 payload로 재생성 가능해야 한다.

### 4.5. Profile Preset Avatar 확장안 (2026-03-18)

- **문제**: 현재 프로필은 고정 아이콘만 사용하고 있어, 사용자가 팀 내에서 자신을 구분하거나 서비스에 애착을 느낄 수 있는 개인화 수단이 부족하다.
- **왜 지금 필요한가**: WIG는 팀 대시보드와 개인 프로필에서 사용자의 존재감이 반복적으로 노출되는 구조다. 닉네임만으로는 시각적 식별력이 약하고, 고정 아이콘은 제품의 완성도를 낮춰 보이게 만든다.
- **운영 제약**:
  - Cloudflare R2 등 별도 object storage를 사용할 수 없다.
  - 사용자 업로드 이미지는 저장/가공/삭제 비용이 커서 현재 운영 범위에 맞지 않는다.
- **MVP 방향(옵션 1 채택)**:
  - 사용자는 직접 이미지를 업로드하지 않고, 서비스가 제공하는 preset avatar 중 하나를 선택한다.
  - 아바타 원본은 외부 스토리지가 아니라 레포 내 정적 자산 또는 코드 생성 가능한 SVG asset으로 관리한다.
  - DB와 API에는 실제 이미지 URL 대신 `avatarKey`만 저장한다.
  - 프론트엔드는 `avatarKey`를 정적 asset 경로 또는 SVG 렌더링 규칙에 매핑한다.
- **브랜드/디자인 원칙**:
  - `docs/design/2026.03.09-branding-identity.md`의 Linear Light Aesthetic를 따른다.
  - 프로필 아바타는 사진 대체재가 아니라 "정교하고 빠른 도구 안의 식별 토큰"으로 설계한다.
  - 과한 캐릭터성보다 1px 라인, 적은 색상 수, 선명한 실루엣, 밝은 배경에서의 식별성을 우선한다.
  - 기본 팔레트는 `Primary (Indigo)`, 중성 그레이, 제한적 포인트 컬러만 사용한다.
- **아이콘 생성 전략(MVP)**:
  - 랜덤 생성형보다는 디자이너가 통제 가능한 8~16종 preset 세트를 먼저 정의한다.
  - 각 아이콘은 동일한 캔버스, 동일한 외곽 비율, 동일한 stroke 규칙을 공유해야 한다.
  - 포맷은 `svg` 우선으로 하여 용량을 줄이고 light UI에서 선명도를 유지한다.
  - 스타일 축은 `shape`, `hair/head`, `eyes`, `accent`, `background ring` 정도의 제한된 조합 규칙으로 설계한다.
  - 생성 방식은 "완전 자동 생성"보다 "브랜드 규칙이 반영된 반자동 템플릿 제작"을 우선한다.
- **데이터/계약 원칙**:
  - `users`에 `avatarKey` 필드를 추가한다.
  - 기존 회원 데이터와 초기 가입 상태를 고려해 `avatarKey`는 `null` 허용으로 시작한다.
  - `GET /api/users/me` 응답에는 `avatarKey`를 포함한다.
  - `PUT /api/users/me`는 `nickname`과 함께 `avatarKey` 수정도 허용하며, `null`로 되돌리는 동작도 허용 여부를 별도 결정한다.
  - 허용 가능한 `avatarKey` 목록은 서버 validation으로 제한하고, `null`은 별도 유효한 값으로 취급한다.
- **UX 원칙**:
  - 프로필 화면에서 아바타 선택은 닉네임 수정과 같은 수준의 가벼운 설정이어야 한다.
  - 선택 UI는 작은 썸네일 그리드로 제공하고, 현재 선택 상태가 즉시 식별되어야 한다.
  - 팀 대시보드, 프로필 상단, 향후 댓글/알림 UI가 생기더라도 같은 `avatarKey`를 재사용한다.
  - `avatarKey`가 `null`인 사용자는 기존 고정 fallback 아이콘을 계속 사용한다.
- **MVP 제외 범위**:
  - 사용자 이미지 업로드
  - 외부 avatar provider 연동
  - 조합형 실시간 avatar builder
  - workspace/role별 전용 스킨
  - 애니메이션 avatar

### 4.6. 인증 전략 정리: 폐쇄형 유지 + 공개 전환 준비 (2026-03-18)

- **현재 판단**:
  - 서비스는 아직 나와 지인들이 사용하는 소규모 폐쇄형 운영 단계다.
  - 따라서 지금 당장 공개 회원가입(Self-signup)과 대형 운영 백오피스를 만드는 것은 우선순위가 아니다.
  - 대신 현재의 폐쇄형 인증을 안정화하고, 나중에 공개 서비스로 전환할 수 있도록 구조적 준비를 해두는 것이 가장 현실적이다.
- **왜 지금 필요한가**:
  - 현재 문서와 구현은 `관리자가 계정을 직접 생성`하는 전제를 이미 강하게 깔고 있다.
  - 반면 실제 구현에는 관리자 권한 체크 누락, 멤버 관리 UI 부재 등 "폐쇄형 운영에 필요한 최소 완성도"가 아직 부족하다.
  - 이 상태에서 공개 서비스용 인증까지 한 번에 열면 제품 검증보다 인증/운영 리스크 대응이 더 큰 일이 된다.
- **제품 원칙**:
  - 현재 운영 모드는 `폐쇄형`으로 유지한다.
  - 현재 제품에 필요한 관리 기능은 `전역 어드민 패널`이 아니라 `워크스페이스 운영자 기능`으로 제한한다.
  - 공개 전환 준비의 핵심은 "어드민을 크게 만드는 것"이 아니라 "인증 정책을 교체 가능하게 만드는 것"이다.
- **지금 해야 하는 범위 (MVP)**:
  - 로그인, 로그아웃, 세션 유지, 비밀번호 변경 흐름을 안정화한다.
  - `POST /api/admin/users`에 실제 관리자 권한 체크를 넣는다.
  - 워크스페이스 멤버 조회/추가/퇴출 같은 최소 운영 기능을 완성한다.
  - 마지막 관리자 보호, 관리자 자기 퇴출 금지, 탈퇴 시 고아 워크스페이스 방지 규칙을 구현 기준으로 확정한다.
- **지금 하지 않을 범위**:
  - 공개 회원가입
  - 소셜 로그인
  - 이메일/휴대폰 인증
  - 전역 관리자 CMS
  - 대규모 고객지원형 운영 콘솔
- **공개 전환을 위해 지금 준비할 구조**:
  - 인증 인프라와 가입 정책을 분리한다.
  - `폐쇄형 사용자 생성`과 `향후 self-signup`이 같은 User 생성/프로필 초기화 흐름을 재사용할 수 있게 서비스 경계를 정리한다.
  - 세션, 프로필, 워크스페이스 가입 후 흐름은 유지하고, "누가 계정을 만들 수 있는가"만 정책으로 분리한다.
  - 공개 전환 시 필요한 보안 항목은 별도 체크리스트로 관리한다.
- **공개 전환 시 후속 과제 (Post-MVP 후보)**:
  - self-signup
  - 비밀번호 재설정
  - rate limiting
  - abuse / spam 대응
  - workspace 생성 권한 정책 재정의

- **실패**: 점수판을 보고도 현재 내가 잘하고 있는지(Winning) 즉각적으로 파악되지 않을 때.

## 8. 구현 메모 (2026-03-15)

- `dashboard/my`는 현재 프로토타입 UI를 유지한 채 실제 세션 쿠키 기반 API를 사용한다.
- 사용자 닉네임 표시는 세션 기반 API 조회 결과를 기준으로 표시한다.
- 변경 파일 기준 ESLint는 통과했다.
- 저장소 전체 `yarn tsc --noEmit`는 이번 작업과 별개인 기존 타입 오류로 아직 통과하지 않는다.

## 9. 기간 선택형 대시보드 성공 기준 (2026-03-16)

- 사용자가 `dashboard/my`에서 임의 날짜를 골라 해당 주 또는 해당 월의 지표를 즉시 확인할 수 있다.
- 주간 모드에서는 어떤 날짜를 고르더라도 결과가 항상 1일치가 아닌 주간 7일 테이블로 일관되게 표시된다.
- 상단 요약 카드와 본문 테이블/리스트가 항상 같은 기간 기준으로 동기화된다.
- URL 공유 또는 새로고침 이후에도 선택한 기간과 보기 단위가 유지된다.
- 월간 모드에서 "이번 달 달성률"이 아니라 "선택한 월 달성률"로 의미가 명확하게 전달된다.

## 10. Preset Avatar 성공 기준 (2026-03-18)

- 사용자는 업로드 없이도 프로필에서 준비된 avatar 중 하나를 1회 선택하고 이후 일관되게 자신의 식별 이미지로 사용한다.
- `avatarKey`만으로 프로필 화면과 팀 대시보드에서 동일한 시각 결과가 재현된다.
- avatar asset은 브랜드 문서의 light/minimal 원칙을 따르며, 작은 크기에서도 서로 구분 가능하다.
- object storage 없이도 자산 관리와 배포가 가능하고, 운영자가 preset 추가/교체를 제어할 수 있다.

## 11. 인증 전략 성공 기준 (2026-03-18)

- 현재 운영 단계에서는 지인 중심 폐쇄형 사용이 불편 없이 유지된다.
- 관리자용 계정 생성 API와 워크스페이스 운영 기능이 최소한의 권한 규칙 아래 안전하게 동작한다.
- 추후 공개 전환 시 세션/프로필/워크스페이스 흐름을 버리지 않고 가입 정책만 확장할 수 있다.
- 공개 회원가입 관련 요구사항이 즉흥적으로 섞이지 않고 별도 마일스톤과 체크리스트로 관리된다.
