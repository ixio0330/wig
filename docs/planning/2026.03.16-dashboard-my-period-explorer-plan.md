# `dashboard/my` 기간 탐색 확장안

## 1. 배경 및 맥락 (Background & Context)

- 기존 `dashboard/my`의 핵심 점수판은 항상 현재 주만 보여주기 때문에, 사용자가 특정 주를 회고하거나 월간 추세를 한 번에 확인하기 어렵다.
- 상단 요약 카드에 주간/월간 달성률을 함께 보여주기 시작하면, 본문 테이블과 요약의 시간 기준이 분리되어 해석이 어려워질 수 있다.
- 따라서 기간 선택은 단순 편의 기능이 아니라, 대시보드 전체 정보 구조를 일관되게 만드는 기반 기능이다.

## 2. 우선순위 및 당위성 (Priority & Rationale)

- 현재 제품은 기록 결과를 잘 쌓는 단계에서, 기록을 읽고 해석하는 단계로 확장되고 있다.
- 특정 기간을 선택해 다시 보는 기능이 없으면, 사용자는 "이번 주" 외의 맥락을 제품 안에서 해석할 수 없다.
- 우선 범위는 `dashboard/my`에만 한정하고, 팀 뷰는 기존처럼 현재 주 중심으로 유지한다.

## 3. MVP 범위 (MVP Scope)

- 사용자가 기준 날짜를 선택할 수 있어야 한다.
- 선택된 날짜를 기준으로 `주간` 또는 `월간` 모드를 전환할 수 있어야 한다.
- 주간 모드에서 날짜 선택은 "그 날짜가 속한 주"를 고르는 용도여야 하며, 결과는 항상 ISO week(월~일) 7일 단위로 보여준다.
- 월간 모드에서는 해당 월 전체를 기준으로 월간 달성률과 선행지표별 집계표를 보여준다.
- 월간 모드에서는 일별 토글 테이블 대신 월 집계 중심 UI를 우선 적용하고, 월 내 일별 상세 drill-down은 후속 범위로 둔다.

## 4. 사용자 경험 원칙 (UX Principles)

- 기본 진입은 여전히 "오늘이 속한 주"다.
- 주간 모드에서는 일자를 하나 선택하더라도 단일 날짜 상세 뷰로 내려가지 않는다.
- 사용자가 과거/미래 주를 탐색해도 "오늘" 강조는 현재 실제 날짜에만 유지한다.
- 기준 날짜와 보기 단위(`week` / `month`)는 URL query에 반영해 새로고침과 공유 시 상태를 복원할 수 있어야 한다.
- 월간 모드에서 월 단위 지표와 주간 전용 지표를 혼합해 오해를 만들지 않도록, 지표 period에 맞는 집계 기준을 명시한다.

## 5. 영향 도메인 (Domain Impact)

- `dashboard/my` 상단 기간 선택 UI
- 주간/월간 조회 파라미터 정규화 로직
- 주간 기록판과 월간 집계표 전환 구조

## 6. 성공 및 실패 기준 (Success & Failure Criteria)

### 성공

- 사용자가 선택한 기간 기준으로 대시보드 전체 정보를 일관되게 해석할 수 있다.
- 주간 모드와 월간 모드의 차이가 UI와 데이터 의미 모두에서 분명하다.
- 새로고침과 공유 후에도 선택 상태가 복원된다.

### 실패

- 날짜 선택이 단일 일자 조회처럼 오해된다.
- 상단 요약과 본문 표의 시간 기준이 다르게 느껴진다.
- 과거/미래 탐색에서 오늘 강조가 잘못 보인다.
