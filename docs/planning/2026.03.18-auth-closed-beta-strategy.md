# 인증 전략 정리: 폐쇄형 유지 + 공개 전환 준비

## 1. 현재 판단 (Current Decision)

- 서비스는 아직 지인 중심의 소규모 운영 단계다.
- 따라서 지금 당장 공개 회원가입과 대형 운영 백오피스를 만드는 것은 우선순위가 아니다.
- 대신 현재의 폐쇄형 인증을 안정화하고, 나중에 공개 서비스로 전환할 수 있도록 구조적 준비를 해두는 것이 더 현실적이다.

## 2. 왜 지금 필요한가 (Why Now)

- 당시 문서와 구현은 `관리자가 계정을 직접 생성`하는 전제를 강하게 깔고 있었다.
- 반면 실제 구현에는 관리자 권한 체크 누락, 멤버 관리 UI 부재 등 폐쇄형 운영에 필요한 최소 완성도가 부족했다.
- 이 상태에서 공개 서비스용 인증까지 한 번에 열면 제품 검증보다 인증/운영 리스크 대응이 더 큰 일이 된다.

## 3. 제품 원칙 (Product Principles)

- 현재 운영 모드는 `폐쇄형`을 기본으로 둔다.
- 현재 제품에 필요한 관리 기능은 전역 어드민 패널이 아니라 `워크스페이스 운영자 기능`으로 제한한다.
- 공개 전환 준비의 핵심은 어드민을 크게 만드는 것이 아니라, 인증 정책을 교체 가능하게 만드는 것이다.

## 4. MVP 범위 (MVP Scope)

- 로그인, 로그아웃, 세션 유지, 비밀번호 변경 흐름을 안정화한다.
- `POST /api/admin/users`에 실제 관리자 권한 체크를 넣는다.
- 워크스페이스 멤버 조회, 추가, 퇴출 같은 최소 운영 기능을 완성한다.
- 마지막 관리자 보호, 관리자 자기 퇴출 금지, 탈퇴 시 고아 워크스페이스 방지 규칙을 구현 기준으로 확정한다.
- 워크스페이스 이름 수정처럼 운영에 자주 필요한 작은 설정 기능을 우선 보강한다.

## 5. 지금 하지 않을 범위 (Non-Goals)

- 공개 회원가입
- 소셜 로그인
- 이메일/휴대폰 인증
- 전역 관리자 CMS
- 대규모 고객지원형 운영 콘솔

## 6. 공개 전환을 위해 지금 준비할 구조 (Preparation for Public Transition)

- 인증 인프라와 가입 정책을 분리한다.
- `폐쇄형 사용자 생성`과 `향후 self-signup`이 같은 User 생성/프로필 초기화 흐름을 재사용할 수 있게 서비스 경계를 정리한다.
- 세션, 프로필, 워크스페이스 가입 후 흐름은 유지하고, "누가 계정을 만들 수 있는가"만 정책으로 분리한다.
- 공개 전환 시 필요한 보안 항목은 별도 체크리스트로 관리한다.

관련 체크리스트:

- `docs/planning/2026.03.18-public-transition-checklist.md`

## 7. Post-MVP 후보 (Post-MVP)

- self-signup
- 비밀번호 재설정 추가 수단
- rate limiting
- abuse / spam 대응
- workspace 생성 권한 정책 재정의
- 첫 사용자 온보딩 재설계
- 공개 가입 사용자와 기존 폐쇄형 사용자 공존 migration 규칙 정의
